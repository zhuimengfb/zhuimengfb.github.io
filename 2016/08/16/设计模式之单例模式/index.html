设计模式之单例模式

Posted on 2016-08-16   |  

单例模式算是众多设计模式中较为简单的一种，其用途就像其名字一样，是希望在一个程序中只生成一个实例并且在全局中都能轻松访问到。要实现一个单例模式，最重要的就是保证在程序中只生成一个该实例，因此肯定需要私有化其构造函数，避免大量创造出实例。根据实现方式的不同，比较常见的有6种方式来实现单例模式，不过其中一些方式都存在一些不足和隐患。

饿汉式

所谓饿汉式是指程序启动时，进行类加载的时候就生成该唯一实例，并且由于其构造函数的私有化，其他类就不能生成该类的新实例，这样就在一定程度上保证了实例的唯一性。简单看看代码吧。

public class Singleton {

      private static final Singleton singleton = new Singleton();

      private Singleton() {
      }

      public static Singleton getSingleton() {
        return singleton;
      }
}

程序在加载Singleton类时，会直接初始化singleton实例，然后由于构造函数为private，因此只能通过getSingleton方法来获取这唯一实例。饿汉式的好处是代码很简单，但也存在一些不足，由于其在程序启动的时候进行初始化，所以不论其是否用到，在内存中都存在该实例，不但影响程序的初始化速度，还影响程序内存的使用。针对这个问题，才出现了懒汉式方法。

懒汉式

懒汉式则是指在需要用到该实例的时候才会去生成这唯一实例，这样就不会影响程序的启动速度，还能在一定程度节约内存。先看代码吧。

public class Singleton {

      private static Singleton singleton;

      private Singleton() {
      }

      public static Singleton getSingleton() {
        if (singleton == null) {
              singleton = new Singleton();
        }
        return singleton;
      }
}

该方法核心是在调用getSingleton方法时判断singleton是否为空，当为空时创建一个，否则直接返回该实例。因此该方法只会在第一次调用时生成实例，之后再次调用则会返回生成的实例，因此这种方式的效率比饿汉式更高。但试想这样一个场景，如果两个线程同时访问getSingleton方法，这时就会创建两个singleton实例，因此该方法并不是线程安全的。针对这个问题，提出了同步懒汉式方式。

同步懒汉式

所谓同步懒汉式就是使得getSingleton方法为同步方法，这样就保证在同一时刻只有一个线程能访问到该方法，从而实现线程安全。代码如下

public class Singleton {

      private static Singleton singleton;

      private Singleton() {
      }

      public static synchronized Singleton getSingleton() {
        if (singleton == null) {
              singleton = new Singleton();
        }
        return singleton;
      }
}

其实就是在懒汉式的前提下把getSingleton方法变为同步而已。但总所周知的是同步会使得方法的效率大大降低，这种方式在任何情况下的访问都是同步式访问，导致程序效率大打折扣。因此，针对这个问题，诞生了双重检查懒汉式。

双重检查懒汉式

由以上知道，只有在第一次访问getSingleton方式时存在线程安全隐患，如果实例已经创建好了，及时线程同时访问也不会产生新的实例，因此我们只需要保证在第一次创建实例的时候线程安全即可。先看代码吧。

public class Singleton {

      private static volatile Singleton singleton;

      private Singleton() {
      }

      public static Singleton getSingleton() {
        if (singleton == null) {
              synchronized (Singleton.class) {
                if (singleton == null) {
                      singleton = new Singleton();
                }
              }
        }
        return singleton;
      }
}

当两个线程同时访问到getSingleton方法时，进行第一次判空之后，由于同步块的作用，同步块里面的代码只有一个线程能访问到，这时再进行一次判空处理，如果为空，则生成实例。注意在这种方式中，singleton实例加了一个volatile属性，该属性能保证一个线程修改该实例之后会立马写回内存。因此当第一个线程访问结束后，将生成一个实例，并且实例会写回主内存中，这时第二个线程进入同步块，进行判空处理的时候发现该实例已经不为空，则不会进行创建。因此该方式只会在第一次创建实例时用同步块的方式解决线程安全隐患，相比较于同步方法方式效率更高。

内部类方式

所谓内部类方式是指将需要的单例类以内部类的方式来创建。我们知道，内部类只有在调用的时候才会初始化。简单看看代码

public class Singleton {

    private Singleton() {
    }

    public static Singleton getSingleton() {
        return SingletonHolder.singleton;
    }

    private static class SingletonHolder {
        private static Singleton singleton = new Singleton();
    }
}

在Singleton类加载的时候，并不会初始化Singleton对象，只有在调用getSingleton方法时，才会加载内部类并且初始化Singleton对象，实现懒加载。

存在的问题

以上提到的方式都不能避免同一个问题，即防反射。即使构造方法都声名为private，Java中的反射机制依然能够调用该构造方法来生成实例。加入以下测试代码

public static void main(String[] args) {
    System.out.println(Singleton.getSingleton().toString());
    try {
        Constructor constructor = Class.forName("Singleton").getDeclaredConstructor();
        constructor.setAccessible(true);
        Singleton singleton = (Singleton) constructor.newInstance();
        Singleton singleton2 = (Singleton) constructor.newInstance();
        System.out.println(singleton.toString());
        System.out.println(singleton2.toString());
    } catch (Exception e) {
        e.printStackTrace();
    }
}

为了方便里面简单处理了异常。从打印台可以看出三个实例的属于同一个类，但hashcode值并不相同，因此，是属于不同的对象。

Singleton@60e53b93
Singleton@5e2de80c
Singleton@1d44bcfa

所以以上方式都不能防止Java通过反射机制来调用构造函数创建更多的实例。

网上有人说用过设置一个标志位，使得构造函数在第二次调用的时候抛出异常阻止生成新的实例。简单代码如下

public class Singleton {

    private static boolean flag = false;

    private Singleton() {
        if (!flag) {
            flag = true;
        } else {
            try {
                throw new Exception("单例已经创建,不能创建第二个单例");
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }

    public static Singleton getSingleton() {
        return SingletonHolder.singleton;
    }

    private static class SingletonHolder {
        private static Singleton singleton = new Singleton();
    }
}

的确，这种方式在一定程度上能解决创建第二个实例的问题，然而，通过反射我们依然可以找到该类申明的变量，即flag，然后通过改变该标志位依然能实现创建第二个单例。所以网上的这种方法不是很靠谱。真正实现防反射还是得依靠jdk1.5引进的枚举方式。

枚举

枚举的出现使得单例模式有了新的方向，因为枚举是天然的单例，并且既能防止反序列化，又能防止Java的反射。因此，用枚举方式来实现单例模式是最安全的，代码也相对简单。这里就不写代码了。

其实在实际开发中，枚举类来实现单例模式用得相对较少，由于枚举在jdk1.5之后才引入的，所以这种写法相对有点陌生。此外，虽然其安全性很高，但其占用的内存会相对其他方法更多，由于一般在程序中很难遇到反射或者反序列化，因此只需要保证线程安全的前提下来实现单例模式即可。所以平常使用得更多的方式还是双重检查锁方式来实现单例模式。
