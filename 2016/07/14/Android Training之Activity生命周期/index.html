Android Training之Activity生命周期

Posted on 2016-07-14   |  

阅读完这篇文章大概需要3分钟。神图镇楼

Activity作为Android四大组件之一，其最基本也是最重要的特征就是其生命周期，只有将Activity的生命周期彻底了解清楚了，才能开发出优质的应用，其重要性可以类比于建筑物的地基一样。

基本生命周期

如神图所示，Activity的生命周期主要有以下几个阶段：

  1. onCreate：这是Activity创建的第一步，即新建一个Activity都会经历这一步。在这个阶段主要完成Activity与界面的绑定，即setContentView方法，对一些成员变量进行初始化操作。
  2. onStart：当Activity对该方法回调时，表明Activity已经可见。只是用户还看不到而已。
  3. onResume：执行该方法时，Activity已经处于前台，可以和用户进行交互。
  4. onPause：暂停阶段，即Activity正在停止。在该回调函数中，一般执行一些不太耗时的操作，比如停止Activity的动画，释放一些资源比如广播监听器等，保存一些轻量级的数据，以便用户回到Activity时能重现出来。
  5. onStop：停止阶段，该阶段的Activity完全不可见，例如用户打开了第二个Activity或者回到了主页面，当前Activity将被完全覆盖而进入该回调函数。在该阶段可以执行一些轻量级的回收工作。
  6. onDestroy：销毁阶段，执行该阶段表明这个Activity将会被销毁。这是Activity生命周期的最后一个阶段，在该方法中我们应该回收该Activity中占用的系统资源。
  7. onRestart：重启阶段，这是生命周期中比较特殊的一个阶段，前六个阶段可以说是一一对应，比如onCreate对应onDestroy，一个代表新生，一个代表毁灭，onStart和onStop前者代表开始，后者代表停止，onResume和onPause则介于开始于停止之间。而onRestart则没有配对的周期，该周期只有在Activity重启时才会执行。比如应用从桌面切换回Activity或者从第二个Activity回到之前的Activity。

以上便是Activity的七个生命周期，对于一个Activity的生命周期的情况，大致有以下几种。

  1. 第一次启动：onCreate->onStart->onResume
  2. Activity从前台失去焦点，但处于可见：onResume->onPause
  3. Activity从前台失去焦点，完全不可见：onResume->onPause->onStop
  4. Activity处于后台，但部分可见切换回前台，比如对话框消失：onPause->onResume
  5. Activity处于后台，完全不可见切换回前台，比如从桌面重新进入界面：onStop->onRestart->onStart->onResume
  6. Activity从前台按返回键退出：onPause->onStop->onDestroy
  7. Activity本身处于后台，由于系统内存不足被回收：onStop->onDestroy

新学者可以通过打印log信息来模拟以上情况，然后在logcat中观察生命周期回调函数的执行情况，简单的打印方法如下

protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);
    Log.d("MainActivity", "onCreate");
  }

同样的方式分别在onStart等其他生命周期中写上log信息然后进行模拟即可。把这些生命周期弄懂之后就可以作为一个入门的Activity使用者了。

Activity生命周期进阶

不知道细心的读者有没有注意到前面提到的在onPause和onStop中都只能做轻量级的资源回收工作。这样做肯定是有道理的。想想在什么情况下这两个阶段才能执行呢。无非就是有新的界面来遮挡或者覆盖当前界面的时候才会执行这两个阶段。那么用户肯定是希望新界面能够快点显示出来。我们知道，这些Activity的生命周期都是在主线程中执行，即都在UI线程。那么新Activity的onCreate等周期和旧Activity的onPause等周期的执行顺序是怎样的呢。可以用代码来实测以下。第一个Activity的代码如下

public class MainActivity extends AppCompatActivity {

  private Button btJump;

  @Override
  protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);
    Log.d("MainActivity", "onCreate");
    btJump = (Button) findViewById(R.id.bt_jump);
    btJump.setOnClickListener(new View.OnClickListener() {
      @Override
      public void onClick(View v) {
        Intent intent = new Intent();
        intent.setClass(MainActivity.this,SecondActivity.class);
        startActivity(intent);
      }
    });
  }

  @Override
  protected void onStart() {
    super.onStart();
    Log.d("MainActivity", "onStart");
  }

  @Override
  protected void onResume() {
    super.onResume();
    Log.d("MainActivity", "onResume");
  }

  @Override
  protected void onPause() {
    super.onPause();
    Log.d("MainActivity", "onPause");
  }

  @Override
  protected void onStop() {
    super.onStop();
    Log.d("MainActivity", "onStop");
  }

  @Override
  protected void onDestroy() {
    super.onDestroy();
    Log.d("MainActivity", "onDestroy");
  }
}

该界面中放置一个按钮，界面初始化后点击按钮跳转到第二个界面，通过logcat可以看出两个Activity的生命周期的执行情况，如下图所示

可以看出当第一个Activity执行了onPause之后，第二个Activity才开始执行onCreate,并且直到界面完全显示，然后再执行onStop。为了使得新界面能尽快显示，我们不能在旧的Activity的onPause中执行耗时操作，同样，为了在新界面中能够响应用户的交互，我们也不能在旧Activity的onStop中执行耗时操作堵塞UI线程。

Activity生命周期异常情况

以上讨论的都是正常情况下的Activity生命周期，然而在系统运行中总会存在一些异常情况。例如系统配置改变会杀死当前Activity并进行重绘，系统内存不足会回收一些Activity。
当系统配置改变比如旋转屏幕，改变系统语言等，会杀死当前Activity并且重新生成，由于是异常结束Activity生命周期，该Activity的一些数据如果不进行保存的话，重新生成将会和第一次进入该界面一样。细心的读者应该会发现在生命周期中，只有onCreate方法带有一个Bundle类型的参数。没错，该参数就是用来保存界面的数据的。当第一次进入界面，该参数理论上为null，而由于系统配置变化重新生成Activity时，该参数一般不为null。最为重要的是，当Activity在这种情况时，在被杀死之前，会自动调用onSaveInstanceState方法，在该方法中我们便可以进行自定义的数据保存工作。而一些系统控件比如EditText上的文字，系统会自动帮我们保存。在新建Activity时，我们可以在onCreate方法中进行数据恢复，也可以在onRestoreInstanceState方法中进行数据恢复。在这里推荐使用第二种方法，因为如果在onCreate中进行数据恢复时，我们需要对Bundle类型的参数进行判空处理，而在onRestoreInstanceState方法，参数肯定不为空。
这时，有人问了，那么我们可以在系统配置变化时不进行Activity重建么？答案是肯定的，我们只需要在AndroidManifest文件中的Activity申明处加上android:configChanges属性即可，比如在旋转屏幕：orientation，屏幕尺寸：screenSize，多个属性可以用“|”隔开。这样的话当屏幕旋转时不会销毁当前Activity，即不会回调onSaveInstanceState方法和onRestoreInstanceState方法，取而代之的是回调onConfigChanged方法。
当系统内存不足时，会自动回收一些优先级较低的Activity，这些Activity一般处于后台，后续同样是通过onSaveInstanceState和onRestoreInstanceState方法来进行数据存储与恢复。

数据恢复的高级用法

我们知道数据恢复时是以Bundle类型来保存的，而Bundle类型只能保存一些基本类型以及一些能够序列化和反序列化的数据，关于序列化和反序列化大家可以查一下，这里就不详细说明了，简言之就是实现了Serializable或者Parcelable借口的数据，如果无法序列化的数据也就无法保存。此外，Bundle也不能保存大数据，因此想要保存非序列化数据或者大数据就需要想其他方法了。
在Android3.0之后，Android提供了一种轻量级的“活动”Fragment，称之为碎片。因为在系统配置改变时，并不会回收Fragment，因此通过Fragment我们可以保存很多数据。具体步骤如下所示：

  1. 扩展 Fragment 类并声明对有状态对象的引用。
  2. 在创建片段后调用 setRetainInstance(boolean)。
  3. 将片段添加到 Activity。
  4. 重启 Activity 后，使用 FragmentManager 检索片段。

例如一个简单的保存数据的Fragment可以像如下代码所示

public class RetainedFragment extends Fragment {

    private MyDataObject data;

    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setRetainInstance(true);
    }

    public void setData(MyDataObject data) {
        this.data = data;
    }

    public MyDataObject getData() {
        return data;
    }
}

而在Activity中，声名一个RetainedFragment对象，然后在onCreate中添加片段或者恢复片段。

public class MyActivity extends Activity {

    private RetainedFragment dataFragment;

    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.main);
        FragmentManager fm = getFragmentManager();
        dataFragment = (DataFragment) fm.findFragmentByTag(“data”);
        if (dataFragment == null) {
            dataFragment = new DataFragment();
            fm.beginTransaction().add(dataFragment, “data”).commit();
            dataFragment.setData(loadMyData());
        }
        // the data is available in dataFragment.getData()
        ...
    }

    @Override
    public void onDestroy() {
        super.onDestroy();
        // store the data in the fragment
        dataFragment.setData(collectMyLoadedData());
    }
}

这里值得注意的是，碎片中一定不能保存对Activity的引用，即不能保存和Context相关的数据，比如drawable，view等，否则，将会引起内存泄漏。所谓的内存泄漏是指本来这个东西逻辑上应该被垃圾回收机制回收掉，但由于存在强引用，垃圾回收机制无法对其进行回收，使得内存无法释放。
嗯，关于Activity的生命周期大概就是这么多内容，掌握好这部分内容是入门Android的前提。
