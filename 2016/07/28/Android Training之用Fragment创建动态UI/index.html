Android Training之用Fragment创建动态UI

Posted on 2016-07-28   |  

Fragment是随着Android 3.0的发布而被人熟知的，随着Android平板电脑的诞生，为Android平板以及手机开发相同的App变得比较困难，毕竟两者的尺寸相差太远。因此Google设计了Fragment这样一个东西来解决开发者的烦恼。并且由于Activity有时候显得过于庞大，项目中Activity太多会显得相当臃肿，以及随着业务逻辑的扩展，如果一味地扩展Activity，项目维护的难度会越来越大，因此，也有人提出了一个Activity+多个Fragment的App架构。总的来说，Fragment有助于创建动态UI，相对于Activity来说更易于扩展。

创建一个简单的Fragment

如果仅仅将Fragment视为Activity的一部分的话，创建Fragment的方式其实和新建一个自定义控件没有太大的差异。所以，简单的步骤如下

  1. 新建一个自定义Fragment类继承于Fragment类。
  2. 在Activity布局文件中申明一个自定义Fragment控件。

为了适配更低版本的Android系统，Google也提供了相关的支持包。默认的Fragment类是Android 3.0之后才有的，所以我们在自定义Fragment时，一般会继承v4包中的Fragment来使得App兼容到Android 1.6。
Fragment和Activity一样拥有自己的生命周期，在自定义Fragment时，一般只需要重写onCreateView方法即可，并且该方法返回Fragment的View即可。代码如下：

public class FirstFragment extends Fragment{

      @Nullable
      @Override
      public View onCreateView(LayoutInflater inflater, @Nullable ViewGroup container, @Nullable Bundle savedInstanceState) {
        return inflater.inflate(R.layout.fragment_first, container, false);
      }
}

然后在Activity的布局文件中申明该自定义控件，值得一提的是，在申明fragment控件时，必须指定唯一的id或者tag属性，或者其父布局具有唯一的id

<fragment
    android:name="com.example.fbi.testapplication.fragments.FirstFragment"
    android:id="@+id/fragment_first"
    android:layout_width="match_parent"
    android:layout_height="match_parent"/>

需要注意的是，Activity的父类必须和Fragment相对应，换句话说，如果想在Activity中加入继承自v4包中的Fragment的自定义Fragment，Activity必须继承FragmentActivity，否则会报错，如果使用的是Android原生的Fragment，则Activity可以随意继承。另外，现在一般项目中都会引入v7 appcompat包，所以继承AppCompatActivity的Activity依然可以显示Fragment，这是因为AppCompatActivity是FragmentActivity的子类。
以上提到的方式只能静态地创建Fragment，这显然没有发挥Fragment的真正实力，因为要创建动态的UI，动态地创建、替换、删除Fragment才是王道，所以接下来讲解动态Fragment相关知识。

动态使用Fragment

在Activity运行时，对Fragment进行添加、替换、删除，可以实现动态改变UI。要实现这个，必须使用到FragmentManager，并且通过事务的方式来实现上述功能，具体步骤可以描述如下：

  1. 通过上下文获取FragmentManager，getSupportFragmentManager(v4包中的方法)或者getFragmentManager方法可以获得和该Activity绑定的唯一FragmentManager。
  2. 创建Fragment事务，可以使用FragmentManager的beginTransaction方法进行创建。
  3. 通过事务进行添加、替换、移除操作。
  4. 对事务提交。

使用动态创建的方式，就可以不用在布局文件中申明fragment控件，只需要申明一个容器布局来存放fragment，我们一般可以使用FrameLayout来作为容器布局。

<FrameLayout
xmlns:android="http://schemas.android.com/apk/res/android"
android:id="@+id/container_layout"
android:layout_width="match_parent"
android:layout_height="match_parent">

</FrameLayout>

而在Activity中，按照如上顺序，即可动态改变Fragment，和之前提到过的类似，Fragment必须和FragmentManager以及FragmentTransaction相对应,如果使用android.app.Fragment类时，只能使用android.app.FragmentManager以及android.app.FragmentTransaction。

public class MainActivity extends FragmentActivity {

      @Override
      protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
           setContentView(R.layout.activity_main);
           FirstFragment firstFragment = new FirstFragment();
           SecondFragment secondFragment = new SecondFragment();
           FragmentManager fragmentManager = getSupportFragmentManager();
           FragmentTransaction fragmentTransaction = fragmentManager.beginTransaction();
        //添加Fragment
        fragmentTransaction.add(R.id.container_layout,firstFragment);
        fragmentTransaction.commit();
        //替换Fragment
        fragmentManager.beginTransaction().replace(R.id.container_layout,secondFragment).commit();
        //移除Fragment
        fragmentManager.beginTransaction().remove(secondFragment).commit();
      }
}

值得一提的是，一旦一个事务提交，即执行了commit之后，该事务就没法继续操作了。而FragmentManager在一个Activity中只会存在一份，即每次调用getSupportFragmentManger都会获得同一个FragmentManager。
另外，在进行Fragment替换时，如果希望按返回键时能够返回到被替换的Fragment，只需在事务中添加addToBackStack方法就行。刚方法传入一个可选的字符串参数，如果不需要改变后台的Fragment顺序，该参数传入为null即可。

fragmentManager.beginTransaction().replace(R.id.container_layout,secondFragment).addToBackStack(null).commit();

Fragment之通信

为了提高UI的复用，我们一般会采用模块化的Fragment，即一个Fragment负责某个模块。一般一个Activity中会有多个Fragment，原则上Fragment之间不要直接通信，更好的做法是通过Activity来进行通信来降低耦合度。比较常用的不同模块通信的方式接口回调。比如如果一个Fragment要和Activity进行通信，我们可以在Fragment中定义接口，在相应地方进行使用，然后在Activity中进行实现该接口即可实现Fragment和Activity的通信。简单一个例子，在Fragment中点击一个按钮，在生成一个随机数，在Activity中得到这个随机数并以Toast方式进行显示。
首先在Fragment类中申明一个RandomNumberListener接口

public interface RandomNumberListener {
       void showRandomNumber(int number);
  }

并且Fragment中持有一个该接口实例，在Fragment和Activity绑定时，初始化该实例。

private RandomNumberListener randomNumberListener;

 @Override
 public void onAttach(Context context) {
      super.onAttach(context);
      randomNumberListener = (RandomNumberListener) context;
 }

在Activity这个实现该接口。

 public class MainActivity extends FragmentActivity implements FirstFragment.RandomNumberListener {

      @Override
      protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
        FirstFragment firstFragment = new FirstFragment();
        FragmentManager fragmentManager = getSupportFragmentManager();
        //添加Fragment
        fragmentManager.beginTransaction.add(R.id.container_layout,firstFragment).commit();
      }

      @Override
      public void showRandomNumber(int number) {
        Toast.makeText(getApplicationContext(), number + "", Toast.LENGTH_LONG).show();
      }
}

然后在Fragment中进行接口函数调用，即可完成Fragment和Activity的通信了。

button.setOnClickListener(new View.OnClickListener() {
  @Override
  public void onClick(View v) {
    randomNumberListener.showRandomNumber(new Random(System.currentTimeMillis()).nextInt());
  }
});

而Activity可以直接获取到Fragment的实例，因此可以直接调用Fragment的共有方法，因此Activity向Fragment通信比较简单。

结合以上两种方式即可实现Fragment和Fragment通信，即Fragment通过接口回调和Activity进行通信，然后Activity获取消息后直接通过其持有的Fragment实例和其他Fragment进行通信，这样就实现了Fragment之间的通信。

Fragment生命周期

和Activity一样，Fragment也有自己的生命周期，并且其生命周期和Activity大部分相同，其生命周期和其绑定的Activity的生命周期基本同步。其生命周期如下图所示

相比于Activity，其生命周期多了几个阶段。Activity回调相应生命周期时，Fragment会同时回调相应的生命周期。除此之外的Fragment生命周期主要是以下几个

  1. onAttach: 在Fragment已与 Activity 关联时调用（Activity 传递到此方法内）。
  2. onCreateView: 调用它可创建与Fragment关联的视图层次结构。
  3. onActivityCreated: 在 Activity 的 onCreate() 方法已返回时调用。
  4. onDestroyView: 在删除与Fragment关联的视图层次结构时调用。
  5. onDetach: 在取消片段与 Activity 的关联时调用。

如果对Activity生命周期比较熟悉的话，Fragment生命周期也就很容易掌握了。

Fragment进阶知识点

在上一节中已经讲过，在Activity异常恢复时，可以借由Fragment来恢复数据，所以Fragment不仅仅可以作为视图加入到Activity中，还可以作为Activity的后台行为添加到Activity中，这里就不再赘述。

在Activity中，对Fragment实例进行管理主要是通过FragmentManager, 通过findFragmentById()（对于在Activity布局中提供UI的Fragment）或 findFragmentByTag()（对于提供或不提供UI的Fragment）获取Activity中存在的Fragment。

对Fragment的改变主要是通过FragmentTransaction进行，并且每次的事务提交之后才会生效。在提交了事务之后，事务并不会立即执行，而是在Activity的UI线程（“主”线程）可以执行该操作时再安排其在线程上运行。不过，如有必要，也可以从UI线程调用executePendingTransactions()以立即执行 commit()提交的事务。

其实Google对Fragment这一块的管理体现出了Java中的单一职责原则，本来可以将Fragment的管理和改变都可以通过FragmentManager来做到，但Google使用了事务这种方式，使得Fragment的管理和改变分开，降低程序的耦合性。另外，FragmentManager和FragmentTransaction都是抽象类，实际上通过getSupportFragmentMangager获取到的实例是FragmentManagerImpl类，其beginTransaction的返回的是BackStackRecord实例，该类继承了FragmentTransaction类，实现了Fragment的add，replace等操作。

Fragment新建实例时，很少通过new的方式来新建。当需要传入参数的时候，传统的想法是通过含参构造函数来实现，然而对于Fragment，当进行重建的时候只会调用默认构造函数，因此会丢失参数。所以建议通过Fragment的setArgument方法和getArgument方法来传递参数，因此大家新建Fragment时，一般会采用如下方式

private static final String ARGUMENT = "argument";

  public static FirstFragment newInstance(String argument) {
       FirstFragment firstFragment = new FirstFragment();
    Bundle bundle = new Bundle();
    bundle.putString(ARGUMENT, argument);
    firstFragment.setArguments(bundle);
    return firstFragment;
  }

此外，Fragment也提供了startActivityForResult方法以及onActivityResult回调，所以Fragment也能实现跳转并且返回结果。然而有点坑的是Fragment并没有提供setResult方法，因此跳转到新的Fragment时，需要借助其绑定的Activity的setResult方法来设置返回数据。其实看完源码后会发现，Fragment中的startActivityForResult最终调用的还是其宿主Activity的startActivityForResult方法，并且宿主Activity中的onActivityReslut回调执行时，会主动执行其Fragment的onActivityResult方法。
