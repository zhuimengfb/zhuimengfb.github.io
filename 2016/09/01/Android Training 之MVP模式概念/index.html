Android Training 之MVP模式浅谈

Posted on 2016-09-01   |  

软件开发大都采用三层架构，分别为表现层、业务逻辑层和数据访问层。表现层主要负责软件的界面以及和用户的交互，业务逻辑层则根据具体的业务对表现层提供数据或者搜集表现层的数据并进行业务处理，而数据访问层则提供对数据的增删改查等操作，是针对裸数据的操作，这里不涉及任何的逻辑。对于Android这类移动应用，应用的重点在于和用户的交互，因为用户体验决定了应用的价值，因此，其核心在于表现层的性能。对于Android开发，目前为人熟知的表现层模式主要有MVC模式、MVP模式和MVVM模式。由于Android开发的特性，即使在不知道MVC模式时，写出的代码便是天然的MVC，所以这里不再详述。而MVVM模式目前还不成熟，因此也暂时跳过，本文主要分析Android开发中的MVP模式。

MVP模式概念

所谓的MVP指的是Model、View和Presenter。在实际的开发中，MVP可能也不能算是纯正的表现层，因为Model作为提供实体数据的模型，有一点数据访问层的味道，而Presenter作为连接View和Model的桥梁，负责数据的逻辑处理，也算有一点业务逻辑层的味道。所以实际上的MVP也算是一个简单的三层经典架构。在Android中，View为UI界面，体现在代码架构中就是各个Activity以及Fragment，负责UI的展示以及和用户的交互，实际我们会抽象出各种接口，然后Activity和Fragment实现接口；Model层主要负责数据的传送，比如从网络获取数据以及提交用户的数据等；Presenter则是单独的逻辑代码，负责连接Model层和View层。三者之间的关系如下图所示

和传统的MVC模式相比，MVP解决了界面和逻辑的耦合，因为MVC模式中，所有的界面代码和逻辑代码都在Activity和Fragment中，这样使得Activity代码耦合度高，显得十分臃肿。而MVP模式中，由于逻辑代码都在Presenter中，Activity和Fragment只需要负责界面交互，这样降低了耦合度，代码逻辑也更加清晰。

MVP模式也有一些缺点，例如在MVP模式中，接口的数量增多，导致代码量增多。此外，Presenter和View相互引用，不做额外的处理的话会导致内存泄漏。当然，在MVP的优势面前，这些缺点已经不能阻碍MVP作为Android开发者十分喜爱的一种开发模式了。

MVP架构搭建

OK，到了最有实际意义的一节了，MVP概念很好理解，但对于Android初学者来说，要写出MVP模式的代码却还是有一定的难度的。这里以用户登录为demo将一步一步完成一个基本的MVP架构搭建。

View层

在Android中，View层主要是界面交互，这些代码理论上应该在Activity或者Fragment中，为了逻辑清晰，我们一般都将其抽象成接口。首先完成用户的登录界面xml，并且在Activity中完成组件的初始化。然后对于登录，其View的接口可以抽象成获取用户输入的用户名、密码，提示登录成功以及登录失败，登录成功，我们需要弹框展示登录用户名。接口代码如下所示

public interface ILoginView {

  String getUserName();

  String getPassword();

  void loginSuccess(String userName);

  void loginFail();

}

然后，Activity实现该接口，Activity代码如下所示

public class LoginActivity extends AppCompatActivity implements ILoginView{

  private EditText userName;
  private EditText password;
  private Button loginButton;

  @Override
  protected void onCreate(@Nullable Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_login);
    initView();
  }

  private void initView() {
    userName = (EditText) findViewById(R.id.et_user_name);
    password = (EditText) findViewById(R.id.et_password);
    loginButton = (Button) findViewById(R.id.bt_login);
  }

  @Override
  public String getUserName() {
    return userName.getText().toString();
  }

  @Override
  public String getPassword() {
    return password.getText().toString();
  }

  @Override
  public void loginSuccess(String userName) {
    Toast.makeText(this, "登录成功," + userName, Toast.LENGTH_SHORT).show();
  }

  @Override
  public void loginFail() {
    Toast.makeText(this, "登录失败" , Toast.LENGTH_SHORT).show();
  }
}

至此，View层的代码基本完成。之所以说基本，是因为还需要配合Presenter稍微添加一点代码，待会儿会提到。

Model层

在Model层，主要接收用户名和密码然后网络传输到服务器进行验证并且返回结果。由于Model层是通过Presenter进行调用，因此，Model层不能主动向Presenter返回结果，因此一般采用回调的方式来返回结果。所以先写一个回调接口。

public interface LoginCallback{
    void loginFail();

    void loginSuccess(String userName);
}

有了该接口后，便可以实现登录的Model层代码，LoginModel的完整代码如下

public class LoginModel {

  public void login(final String userName, final String password, final LoginCallback loginCallback) {
    new Thread(new Runnable() {
      @Override
      public void run() {
        try {
          Thread.sleep(2000);
        } catch (InterruptedException e) {
          e.printStackTrace();
        }
        if ("fbi".equals(userName) && "fbi".equals(password)) {
          loginCallback.loginSuccess(userName);
        } else {
          loginCallback.loginFail();
        }
      }
    }).start();
  }


  public interface LoginCallback{
    void loginFail();

    void loginSuccess(String userName);
  }
}

至此，Model层代码完成。

Presenter层

在Presenter层中，主要需要响应用户点击登录按钮之后将用户名和密码传递到Model层作用户校验，校验完成之后根据校验结果展示不同的界面。根据MVP模式图，Presenter需要连接Model和View，因此Presenter需要持有View的实例和Model的实例。此外，View层需要将数据传递给Presenter，因此View层需要持有Presenter的实例。一般有两种方式来实现Presenter与View的绑定，一种是构造函数的方式，另一种是提供显式的绑定方法。

public LoginPresenter(ILoginView loginView) {
    this.loginView = loginView;
    loginModel = new LoginModel();
}

public void bind(ILoginView loginView) {
    this.loginView = loginView;
    loginModel = new LoginModel();
}

这两种方法选择其中一种在Activity的onCreate方法中进行View和Presenter的绑定。如果选用第一种方式，则在onCreate中直接通过LoginPresenter(this)进行Presenter的初始化和绑定；使用第二种方式，则需要通过LoginPresenter的默认构造函数初始化后调用bind(this)进行绑定。

然后在Presenter完成登录逻辑处理，接收用户名和密码，传递给Model，在回调中得到登录结果并更新View，由于回调一般在新的线程中执行，因此更新View得借助Handler来处理。LoginPresenter的完整代码如下所示。

public class LoginPresenter {

  private ILoginView loginView;
  private LoginModel loginModel;
  private Handler handler;

  public void bind(ILoginView loginView) {
    this.loginView = loginView;
    loginModel = new LoginModel();
    handler = new Handler();
  }

  public void login(String userName, String password) {
    loginModel.login(userName, password, new LoginModel.LoginCallback() {
      @Override
      public void loginFail() {
        handler.post(new Runnable() {
          @Override
          public void run() {
            loginView.loginFail();
          }
        });
      }

      @Override
      public void loginSuccess(final String userName) {
        handler.post(new Runnable() {
          @Override
          public void run() {
            loginView.loginSuccess(userName);
          }
        });
      }
    });
  }
}

最后在Activity中完成点击事件的处理，这样一个简单的MVP模式就完成了。这只是一个最简单的MVP模式，还没有解决内存泄漏的问题，要解决内存泄漏，原理就是解除View和Presenter的绑定，然后在Activity的onDestroy中调用解绑方法，使得Presenter不再持有View的引用，这样系统就可以回收Activity了。最简单的解绑代码如下所示

public void unBind() {
    this.loginView = null;
    this.loginModel = null;
}

然而这样会导致另外一个问题，当用户验证较慢时，返回结果时，用户已经退出登录界面并执行了解绑方法，这是loginView已经为空，导致空指针异常。所以在对View操作前，需要对View进行判空处理。

高级MVP模式架构

传统的MVP中，Presenter和Model通过回调的方式来通信，这样会产生很多Callback接口，并且Callback一般在新线程中执行，不能直接更新View，导致引入Handler的麻烦。此外，解绑后，还得注意空指针异常，所以显得比较繁琐。当RxJava出来之后，配合MVP模式可以实现更加方便高级的用法。使用RxJava，我们可以不再使用任何Callback，并且解绑之后，还可以通过取消订阅的方式停止代码运行。在决定选择MVP模式作为应用的开发架构时，我们可以先实现一个基本架构，方便之后的开发。

在基本架构中，主要是各种Base类，比如BaseActivity，BasePresenter，BaseModel以及BaseView等。在BaseActivity中，调用Presenter的初始化以及解绑方法，界面的初始化，在BasePresenter中，实现和View，Model的绑定，以及和View的解绑。BaseModel和BaseView可以为空，BaseView中也可以提炼出共有的界面方法，比如没有数据时怎么展示，没有网络怎么展示。BaseActivity如下。

public abstract class BaseActivity<P extends BasePresenter> extends AppCompatActivity {

  public P presenter;

  @Override
  protected void onCreate(@Nullable Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    initPresenter();
  }

  public abstract void initPresenter();

  @Override
  protected void onDestroy() {
    super.onDestroy();
    if (presenter != null) {
      presenter.unBind();
    }
  }
}

主要实现和Presenter的绑定和根据生命周期进行解绑。采用泛型的方式，可以避免在Activity中进行类型的强制转换。

BasePresenter代码如下，同样采用泛型来解决类型强制转换问题，并且引入了订阅管理，在解绑时直接取消订阅，避免空指针问题。由于Presenter的初始化工作在View层，因此在构造函数或者bind方法直接传入View接口进行绑定，而Presenter持有Model实例实在Presenter中，因此采用abstract方法让具体的Presenter来生成Model实例，并且在和View进行绑定时执行。最后在解绑时取消订阅以及解绑防止内存泄漏。

public abstract class BasePresenter<V extends IBaseView, M extends BaseModel> {

  public V view;
  public M model;
  public List<Subscription> subscriptions = new ArrayList<>();

  public BasePresenter(V baseView) {
    this.view = baseView;
    initModel();
  }

  public abstract void initModel();

  public void unBind() {
    for (Subscription subscription : subscriptions) {
      if (subscription.isUnsubscribed()) {
        subscription.unsubscribe();
      }
    }
    view = null;
    model = null;
  }
}

当然，使用RxJava的话，不要忘记引入RxJava的依赖。至此，MVP模式基本完成，可以根据这些基类根据具体的业务来构建应用。完整的代码可以参考Github
