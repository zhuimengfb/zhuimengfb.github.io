设计模式之策略模式与简单工厂模式

Posted on 2016-09-02

当看到策略模式和简单工厂模式之后，突然有点傻傻分不清楚。策略模式通过接口方式来实现不同的行为，然后通过实行不同的策略来选择不同的行为。而简单工厂模式通过继承方式创建不同的对象，让客户端通过简单的tag来创建不同的对象。如果看两者的代码，会发现其实框架都差不多，但从其本质来说，一个重在接口，一个重在继承，这就决定了策略模式重在行为选择，而简单工厂模式重在对象创建。

策略模式

假设这样一个场景，一个人需要走一段路，交通工具可以是摩托车，公交或者出租车，每种交通工具的费用随着路程变化而变化，如果客户想要得到某个距离下的某种交通工具的费用，一般代码可以这样写：

public class TransportContext {

    private static final int TYPE_MOTO = 0;
    private static final int TYPE_BUS = 1;
    private static final int TYPE_TAXI = 2;


    public int calculate(int distance, int type) {
        if (type == TYPE_MOTO) {
            return calculateMoto(distance);
        } else if (type == TYPE_BUS) {
            return calculateBus(distance);
        } else if (type == TYPE_TAXI) {
            return calculateTaxi(distance);
        } else {
            return 0;
        }
    }

    private int calculateMoto(int distance) {
        //具体计费方式请忽略。。。。
        return 0;
    }

    private int calculateBus(int distance) {
        //具体计费方式请忽略。。。。
        return 0;
    }

    private int calculateTaxi(int distance) {
        //具体计费方式请忽略。。。。
        return 0;
    }
}

客户端调用时直接调用calculate方法即可。然而，思考这样一个问题，假如这是还可以坐火车的话，不仅要修改TYPE变量，还需要添加else if分支。这样，不仅代码难看，还违反了开闭原则，即对扩展开放，对修改关闭。这时，策略模式就派上用场了。
首先，写一个交通策略接口。

public interface ITransportStrategy {

    int calculate(int distance);
}

具体的交通方式则可以实现该接口。

public class MotoStrategy implements ITransportStrategy {
    @Override
    public int calculate(int distance) {
        //忽略具体计费方式。。。
        return 0;
    }
}

public class BusStrategy implements ITransportStrategy {
    @Override
    public int calculate(int distance) {
        return 0;
    }
}

public class TaxiStrategy implements ITransportStrategy {
    @Override
    public int calculate(int distance) {
        return 0;
    }
}

而在环境类中，则可以大大简化.

public class TransportContext {
    private ITransportStrategy transportStrategy;

    public TransportContext(ITransportStrategy transportStrategy) {
        this.transportStrategy = transportStrategy;
    }

    public int calculate(int distance) {
        if (transportStrategy != null) {
            return transportStrategy.calculate(distance);
        } else {
            return 0;
        }
    }

    public void setTransportStrategy(ITransportStrategy transportStrategy) {
        this.transportStrategy = transportStrategy;
    }
}

这样，客户端调用时，直接传入具体的策略即可得到相应策略的行为，有新增策略时，只需要新增一个策略类即可，环境类和已存的策略完全不需要修改，完全符合开闭原则。

简单工厂模式

再来看看简单工厂模式，在很多情况下，我们新建一个对象都是通过new的方式来创建出对象，然后再调用该对象的方法。然而实际情况是我们基本不关心该对象的创建过程，而是关注该对象的方法。思考这样一个场景，加入一个汉堡店的顾客想知道各种汉堡的价钱，那么前台需要获取到各种汉堡然后调用其price方法来获取价钱。如果想要隐藏对象的创建过程，则可以使用简单工厂方式。

首先是创建汉堡基类，然后各种汉堡继承该基类。

public abstract class Hamburger {

    public abstract int getPrice();
}

public class BeefHamburger extends Hamburger {
    @Override
    public int getPrice() {
        return 0;
    }
}

public class ChickenHamburger extends Hamburger {
    @Override
    public int getPrice() {
        return 0;
    }
}

然后创建一个汉堡工厂

public class HamburgerFactory {

    public static final String TYPE_BEEF = "beef";
    public static final String TYPE_CHICKEN = "chicken";

    public static Hamburger createHamburger(String type) throws Exception {
        if (type == null) {
            throw new Exception("no such hamburger");
        } else if (type.equals(TYPE_BEEF)) {
            return new BeefHamburger();
        } else if (type.equals(TYPE_CHICKEN)) {
            return new ChickenHamburger();
        } else {
            throw new Exception("no such hamburger");
        }
    }
}

这样客户端调用时，则可以通过工厂直接创建相应的汉堡。

Hamburger beefHamburger = HamburgerFactory.createHamburger(HamburgerFactory.TYPE_BEEF);

这样，尤其在创建对象比较麻烦的时候，通过简单工厂方式可以大大简化客户端的操作。

策略模式和简单工厂模式融合

粗略地对比两种方法，会惊奇地发现策略模式不就是为了解决简单工厂模式的if分支以及修改麻烦的问题嘛，既然这样，为啥还需要简单工厂这种方式，修改成策略模式不是更好吗？

简单工厂重在隐藏对象创建过程，策略模式重在简化行为选择。要简化掉简单工厂中的if else分支，那么，我们只能创建不同的工厂方法，为每种具体的产品创建单独的一个工厂。

public interface IHamburgerFactory {

    Hamburger createHamburger();
}

public class BeefHamburgerFactory implements IHamburgerFactory {
    @Override
    public Hamburger createHamburger() {
        return new BeefHamburger();
    }
}

public class ChickenHamburgerFactory implements IHamburgerFactory {
    @Override
    public Hamburger createHamburger() {
        return new ChickenHamburger();
    }
}

这样客户端就可以通过不同的工厂创建不同的产品了，这时融合策略模式。

public class HamburgerContext {

    IHamburgerFactory hamburgerFactory;

    public HamburgerContext(IHamburgerFactory hamburgerFactory) {
        this.hamburgerFactory = hamburgerFactory;
    }

    public void setHamburgerFactory(IHamburgerFactory hamburgerFactory) {
        this.hamburgerFactory = hamburgerFactory;
    }

    public Hamburger createHamburger() {
        return hamburgerFactory.createHamburger();
    }
}

这样客户端就可以通过策略模式来创建出相应的产品。

Hamburger chickenHamburger = new HamburgerContext(new ChickenHamburgerFactory()).createHamburger();

其实这样一弄，代码量增加了好多。可能有人会吐槽，我就是想创建一个汉堡嘛，用得着写这么多代码嘛！这只是一个简单的例子，在很多情况，创建一个对象有可能会十分复杂，后端并不想公开这个对象的创建过程，因此需要工厂方法来隐藏创建对象过程。而随着对象种类的增多，如果一直按照简单工厂模式那样，会高度耦合在一个核心工厂类中，违反开闭原则。因此利用策略模式降低这个耦合，将工厂分散开来，一个工厂只需要负责生产一种产品，然后客户端通过不同的策略来创建出不同的产品，这样有新增产品时，后端只需要增加产品类和产品工厂类就行了。
