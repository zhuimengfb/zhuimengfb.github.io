Android Splash页面优雅跳转方案

Posted on 2017-01-15   |  

在Android应用中，每个App进入时，大都会有一个过渡页面，用来展示品牌信息，或者在该页面做一些数据的初始化工作，然后隔三秒钟的样子进入应用的主页面。有了这个过渡过程，用户打开App虽然延迟了两三秒，但增强了应用的品牌意识，并且打开App不会显得很突兀，不过在此不讨论这个页面的存在意义，而是讨论一个问题。因为一般是三秒钟进入App主页面，所以如果在三秒钟之内触摸了返回键，按理说应该直接退出程序，然而如今很多App依然会在三秒的样子跳到主页面，那么这个问题是怎么产生以及怎么解决呢？请看我下面的胡扯。

一般的SplashActivity跳转

在Android中，页面跳转肯定是用Intent的这点毫无疑问，然后要做延时跳转，大部分开发者会选择Handler的postDelay这个方法，所以传统的SplashActivity页面跳转的代码就显得很简单了，直接上代码。

new Handler().postDelayed(new Runnable() {
  @Override
  public void run() {
    SplashMainActivity.toMainActivity(SplashActivity.this);
    finish();
  }
}, DELAY_PERIOD);

这部分代码放在SplashActivity的onCreate生命周期中，等延迟三秒的样子后，便会自动跳转到主页面。然而，当不到三秒时，选择触摸返回键退出程序时，虽然SplashActivity生命周期已经结束，但由于Handler中的Runnable对象持有了SplashActivity的引用，不仅造成SplashActivity的内存泄漏，还导致Runnable对象会继续执行run方法，从而跳转到主页面。这不仅从程序上有漏洞，对于用户来说，也是一种不好的用户体验。那么这个问题如何结局呢？

Handler实现页面优雅跳转

首先还是用Handler这套思路，首先，为了不引起SplashActivity的内存泄漏，我们需要将Runnable中对Activity的强引用改为弱引用，然后，如果想要在中途退出页面后不进行页面跳转，那么我们就不能让Runnable对象的run方法执行跳转，所以总结出来的步骤如下

  • 重写Handler类并重写其HandleMessage方法
  • 该类包含一个Activity的弱引用属性，写构造方法传入Activity并赋给弱引用属性
  • SplashActivity中设置标志位，当程序退出，即SplashActivity的onDestroy生命周期回调时，改变标志位
  • 在Runnable中根据标志位判断是否发送跳转页面的消息
  • 在重写的Handler类中的HandleMessage方法中，处理跳转逻辑

完整的代码如下所示

final MyHandler myHandler = new MyHandler(this);
myHandler.postDelayed(new Runnable() {
  @Override
  public void run() {
    if (!exist) {
      myHandler.sendEmptyMessage(WHT_TO_MAIN_ACTIVITY);
    }
  }
}, DELAY_PERIOD);

static class MyHandler extends Handler {
    WeakReference<Activity> activityWeakReference;

    private MyHandler(Activity activity) {
      activityWeakReference = new WeakReference<>(activity);
    }

    @Override
    public void handleMessage(Message msg) {
      super.handleMessage(msg);
      Activity activity = activityWeakReference.get();
      if (msg.what == WHT_TO_MAIN_ACTIVITY && activity != null) {
        SplashMainActivity.toMainActivity(activity);
        activity.finish();
      }
    }
  }

值得注意的一点是，在自定义继承Handler的类中，我们使用了static属性，这使得该类不是SplashActivity的内部类而是静态内部类。因为内部类对象会持有外部类的引用，所以为了避免内存泄漏，建议使用静态内部类的方式。用这种方式后，由于有exit标志位的判断，使得在该页面退出后，就不会发送跳转到主页面的消息，因此就不会有烦人的页面跳转了，并且用弱引用的方式解决了内存泄漏问题。

开发者大多会采取这种方式，因为Handler是属于Android特有的方式，那么还有其他方式可以实现该功能么？答案是肯定的，那就是Java提供的Timer类，该类可以实现一些延时任务，所以做延时跳转肯定是没问题的。

Timer实现页面优雅跳转

Timer是Java内置的类，该类相当于一个计时器，然后可以设定任务以及任务执行的时间，在该场景中，任务便是跳转。用该方法，相对于Handler来说，更简单一点，因为Timer不仅可以新建任务，还可以取消任务，这是Handler不具备的，所以，我们大可以在onDestroy生命周期中调用Timer的取消任务方法，这样，在页面退出后，任务就没了，也就无所谓页面跳转了。具体代码也相对简单，如下所示

timer = new Timer();
timer.schedule(new TimerTask() {
  @Override
  public void run() {
    SplashMainActivity.toMainActivity(SplashActivity.this);
    finish();
  }
},DELAY_PERIOD);

@Override
protected void onDestroy() {
  super.onDestroy();
  timer.cancel();
}

timer的初始化和新建任务放在onCreate生命周期中，然后取消任务放在onDestroy生命周期中，这种方式也能实现优雅跳转的功能。那么还有其他的方法么？答案是肯定的。

CountDownTimer实现页面优雅跳转

既然Timer可以实现延时任务，那么Android提供的计时类CountDownTimer当然也可以咯。CountDownTimer其实多用在首页的广告中倒计时以及输入验证码时的倒计时，用在Splash页面的跳转中，也是可以的，当倒计时结束，直接进行跳转，当倒计时还没结束，用户选择退出应用时，CountDownTimer也提供了取消方法，所以用起来和Timer很类似，所以直接上代码。

countDownTimer = new CountDownTimer(DELAY_PERIOD, COUNT_DOWN_INTERVAL) {
  @Override
  public void onTick(long millisUntilFinished) {

  }

  @Override
  public void onFinish() {
    SplashMainActivity.toMainActivity(SplashActivity.this);
    finish();
  }
};
countDownTimer.start();

在onCreate初始化CountDownTimer对象并且在计时结束时进行跳转任务。

@Override
protected void onDestroy() {
  super.onDestroy();
  countDownTimer.cancel();
}

在onDestroy中取消倒计时，免得退出应用后还进行跳转。原理和Timer挺类似的了，所以就不再详述。

那么还有其他方法么？这我就不知道了，也许还有，只是我暂时还没想到或者遇到，所以以后遇到了再来补充。在这里呢，还是推荐用第一和第三种方案，毕竟是Android提供的嘛，肯定更加合适。并且在Splash页面中，也有很多情况是倒计时广告，在倒计时的地方会有倒计时以及跳过按钮，所以用第三种方案会非常简单，而且CountDownTimer中的onTick回调运行在主线程，所以可以直接更新UI，这多方便呀！
